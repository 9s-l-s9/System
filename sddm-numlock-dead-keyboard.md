# The keyboard that wasn't dead: SDDM, NumLock, and a StumpWM session that ignored every keystroke

**TL;DR:** On Guix System, `sddm-service-type` defaults to `Numlock=on` (upstream
SDDM defaults to `none`). The greeter locks the NumLock modifier, the shared X
server carries that locked state into your X11 session, and a window manager
whose keybindings rely on passive key grabs — StumpWM, in my case — can end up
matching *none* of your keystrokes. The result looks exactly like a dead
keyboard: mouse fine, VT switching fine, typing at the login screen fine, and
not a single key doing anything inside the session. The fix is one line:

```scheme
(service sddm-service-type
         (sddm-configuration
          ;; Guix defaults this to "on"; upstream SDDM defaults to "none".
          (numlock "none")
          ...))
```

## The setup

ThinkPad T450s running Guix System, StumpWM as the window manager, and a fresh
migration from GDM to SDDM (I wanted a login screen I could actually theme —
that part worked out fine). After the switch, logging into StumpWM gave me a
session where the mouse pointer moved happily and the keyboard did *nothing*.
Not "wrong characters" nothing — literally nothing. Prefix key, plain letters,
everything ignored.

Logging into a Plasma Wayland session from the same greeter: perfectly fine.
Typing my password *into the greeter itself*: perfectly fine. Same machine,
same keyboard, same X server even.

## The red herrings

This bug hid behind three unrelated problems, each of which was real and
needed fixing, and each of which made the symptom look like something else:

1. **The greeter theme silently fell back to a built-in default.** Guix ships
   a Qt6-only SDDM, and themes must declare `QtVersion=6` in their
   `metadata.desktop`, or the daemon demands a Qt5 `sddm-greeter` binary that
   doesn't exist and quietly uses its embedded theme. All of SDDM's bundled
   themes hit this. So the login screen didn't even look like it was using my
   configuration, which pointed suspicion at everything except the real bug.

2. **My compositor was crashing at session start.** Newer picom requires an
   explicit `--backend`; mine had none and died instantly. A session with no
   compositor and a known SDDM quirk (the greeter's last frame lingering on
   screen) made the desktop look frozen when it wasn't.

3. **StumpWM's config recompiled itself at every login.** An
   `(asdf:load-system "stumpwm")` at the top of my config triggered a full
   SBCL recompile of StumpWM and its dependencies — about 40 seconds on a
   dual-core T450s, during which the session genuinely doesn't respond to
   anything. If you give up during those 40 seconds (I did, twice), you can't
   tell this delay apart from the real bug that comes after it.

Fix all three and the keyboard is *still* dead. Wallpaper up, mode line up,
mouse clicks registering in the mode line — StumpWM demonstrably alive and
interactive — and zero response to any key.

## The diagnostic that cracked it

The trick was to stop guessing and make the session diagnose itself. I added a
one-shot script to StumpWM's init that, 12 seconds after startup, dumped the
connected X clients and input devices, fired *synthetic* keystrokes via XTEST,
and then recorded raw XInput2 events for 15 seconds while I mashed the
keyboard:

```sh
xlsclients -l
xinput list
xdotool key F2; xdotool key F1
timeout 15 xinput test-xi2 --root
```

One login later, the log told the whole story. The keyboard hardware was fine:
138 raw key events, all present and correct. The X server was delivering them:
each one also appeared as a proper `KeyPress`. And every single event carried
the same stamp:

```
modifiers: locked 0x10 latched 0 base 0 effective: 0x10
```

`0x10` is **Mod2 — NumLock**. Locked. On a laptop where I had never touched
NumLock in my life.

That explained everything at once. StumpWM binds keys with *passive grabs*,
which are registered for exact key-plus-modifier combinations. A locked
modifier is silently added to every key event, and in this session StumpWM's
grabs didn't cover the Mod2 variants — so every keystroke sailed past with
nobody listening. The greeter was immune because a text field takes focused
input, not grabs. The mouse was immune because pointer events don't carry
keyboard modifiers. `Ctrl+Alt+F2` was immune because the X server handles VT
switching itself. A perfect imitation of dead hardware.

And where did the locked NumLock come from? `/etc/sddm.conf`, generated by
Guix:

```
Numlock=on
```

Guix's `sddm-configuration` record defaults the `numlock` field to `"on"`.
Upstream SDDM defaults to `none` (leave it alone), and GDM — which I'd been
using for years without issue — never touches it. Since SDDM's greeter and the
X11 session share one X server, the greeter's NumLock survives the login
handoff and lands in your lap.

An interesting footnote: in a clean Xvfb, StumpWM handled NumLock just fine —
it grabs the Mod2 variants like any well-behaved WM. The failure only occurred
in the real greeter-to-session handoff, presumably because StumpWM computes
its modifier map while the keymap is still settling. If you can't reproduce a
bug headlessly, that's data too.

## The fix

System side, turn the default off:

```scheme
(service sddm-service-type
         (sddm-configuration
          (numlock "none")
          ...))
```

And a belt-and-braces line at the top of StumpWM's init, so no display manager
can ever do this to me again (package: `numlockx`):

```lisp
(run-shell-command "numlockx off")
```

Reconfigure, `herd restart display-manager`, log in — keyboard works, bone
layout and all.

## Takeaways

- If your keyboard is "dead" in exactly one WM but works at the greeter and on
  VTs, check the **modifier state** before you suspect drivers, layouts, or
  grabs held by ghost processes. `xinput test-xi2 --root` shows you every raw
  and delivered event, including the modifier stamp, in one command.
- When a bug survives several plausible fixes, instrument the failing session
  to report on itself instead of iterating blind. One automated login attempt
  produced more truth than four evenings of hypotheses.
- If you migrate display managers on Guix, read the generated `/etc/sddm.conf`
  once. The defaults are not upstream's defaults.
