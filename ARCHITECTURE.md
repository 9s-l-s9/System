# Architecture

## Purpose

This repository is a personal Guix-based system configuration.
It manages three related layers:

1. Channel selection for package/module sources.
2. System-level Guix System definitions.
3. User-level Guix Home and dotfiles configuration.

The dominant idea is composition through small Scheme modules rather than one monolithic system file.

## High-Level Layout

- `channels.scm`
  Defines the Guix channels used by `guix pull`. This is the root of the package/module universe the rest of the repo depends on.

- `systems/`
  Contains machine-specific operating system entrypoints such as `T450s.scm` and `X1.scm`, plus the shared base definition in `base-system.scm`.

- `home/`
  Contains user-level home environments, package groups, and reusable home services.

- `scripts/`
  Contains helper scripts and personal workflows launched from the desktop environment or terminal dashboard.

- `assets/`
  Stores static artifacts such as images and keyboard layout data.

## Configuration Flow

The intended workflow is:

1. Run `guix pull --channels=.../channels.scm`.
2. Reconfigure the host with `guix system ... systems/$HOSTNAME.scm`.
3. Reconfigure the user environment with `guix home ... home/$USER-home-configuration.scm`.

This means the repository is evaluated in two separate Guix contexts:

- System evaluation loads modules from `systems/`.
- Home evaluation loads modules from `home/`.

Because of that split, shared concepts are duplicated structurally rather than through one global module tree.

## System Architecture

`systems/base-system.scm` is the common operating-system template.
It defines the shared baseline:

- kernel, firmware, initrd
- base users and groups
- common system packages
- desktop and device services
- Guix service customization
- mDNS name service switching

Each host file in `systems/` inherits from `base-system` and overrides machine-specific facts:

- hostname
- keyboard layout
- root and boot filesystems
- swap devices
- bootloader target

This produces a clear model:

- `base-system.scm` owns policy
- `systems/<host>.scm` owns hardware identity

That separation is the main architectural boundary on the system side.

## Home Architecture

The home side is assembled from three pieces:

### 1. Entry Points

Files such as `home/samuel-home-configuration.scm` build a `home-environment`.
These files decide:

- which package set to install
- which home services to enable
- whether the checked-in dotfiles tree is installed

### 2. Package Groups

`home/packages/base-packages.scm` collects package specifications into semantic groups:

- `programming-packages`
- `web-packages`
- `utilities-packages`
- `xorg-packages`
- `fonts-packages`
- `emacs-packages`
- `kde-packages`
- `app-packages`

`all-packages` is the aggregation point used by the main home configuration.

This file is effectively the package policy layer for the user environment.

### 3. Service Modules

`home/services/*.scm` contains reusable Guix Home service constructors.
Current examples:

- `fish.scm` for shell aliases and environment variables
- `git.scm` for Git identity and defaults
- `stumpwm.scm` for desktop/window manager behavior
- `redshift.scm`, `lem.scm`, `bash.scm`

These modules are the behavior layer.
They encode runtime preferences rather than package membership.

## Dotfiles Model

The repo currently uses `home-dotfiles-service-type` to install the checked-in `home/files` tree directly into the home directory.
That means `home/files/.config/...` is a source of truth for several applications, including Helix.

This is important because there is also an in-progress alternative model:

- `home/services/helix-home-services.scm`
- `home/services/helix.scm`

Those modules aim to generate Helix configuration from Scheme instead of shipping static files.

So the repo is in a transitional state between two patterns:

1. Static dotfiles copied from `home/files`.
2. Declarative config generated by custom home services.

At the moment, the static-dotfiles path is the active workflow because the Helix service is not wired in.

## Desktop and Workflow Integration

The desktop experience is centered around StumpWM and custom scripts.

`home/services/stumpwm.scm` binds commands to scripts in `scripts/`, including:

- `terminal-dashboard.scm`
- `dashboard.scm`
- `add-todo.scm`

This creates a practical workflow architecture:

- Guix defines the environment.
- StumpWM is the launcher/control plane.
- Scripts implement task-specific workflows.

The scripts are not isolated utilities; they are part of the interactive UX of the system.

## Shared Settings

`home/shared-settings.scm` holds cross-cutting personal identity values such as email and username.
This is used by service modules like Git config and avoids repeating those values in multiple service definitions.

Conceptually, this is the smallest shared configuration layer on the home side.

## Process Ideas Behind The Repo

The code suggests a few core process ideas:

- Prefer inheritance for machines.
  Hardware-specific files should stay thin and inherit a stable base.

- Prefer composition for user behavior.
  Home services and package groups are split by responsibility so they can be enabled or changed independently.

- Keep operational commands simple.
  The README documents a direct `pull -> system reconfigure -> home reconfigure` loop rather than a wrapper tool.

- Keep personal workflow inside the repo.
  Window manager actions and helper scripts live beside the system definitions instead of being managed separately.

## Current Friction Points

These are architectural tensions visible in the current tree:

- The home side mixes generated configuration and checked-in dotfiles.
  That is workable, but only if ownership of each target path is explicit.

- Some aliases and paths are inconsistent with current filenames.
  Example: the Fish alias for `gh` points at `home/home-configuration.scm`, while the active entrypoint is `home/samuel-home-configuration.scm`.

- `base-system.scm` is evolving from a StumpWM-only base toward a broader desktop base.
  The addition of Plasma-related services and packages makes the base layer less opinionatedly minimal than before.

- The Helix modules are an architectural experiment that has not yet replaced the existing `home/files/.config/helix/*` source of truth.

## Recommended Mental Model

When changing this repo, treat it as four layers:

1. `channels.scm` selects the ecosystem.
2. `systems/base-system.scm` defines shared machine policy.
3. `systems/*.scm` define host hardware facts.
4. `home/` defines user software, behavior, and dotfiles.

Within `home/`, keep a strict distinction:

- package lists answer "what is installed?"
- service modules answer "how should it behave?"
- `home/files` answers "which apps still use static dotfiles?"

If a tool moves from `home/files` to a generated service, delete or retire the old static target in the same change so ownership stays unambiguous.
